Git commands:

  • git config --global user.name ivan.podavan - налаштовуємо ім*я людини, яка працює з репозиторієм, буде потім показуватись в комітах як ім*я автора комітів
  • git config --global user.name ivan.podavan@gmail.com - налаштовуємо пошту, яка буде показуватись в інформації про коміти, які зробив автор
  • git init [назва репозиторію, без дужок] - створюємо новий репозиторій
  • git status - показує всі нові чи змінені файли, які потрібно зафіксувати (закомітити?)
  • git add [назва файлу, без цих дужок, так це важливо, бо я спочатку не зрозуміла] - індексує файл, який потім будемо комітити
  • git commit - створюємо новий коміт, який включає всі поточні зміни проіндексованих файлів і файл логів
  • git commit -m "текст коментару" - те саме плюс додаємо текст коментару
  • git log - можна подивитись зміни файлів проєкту
  • git checkout [назва вітки] - переключаємось на вибрану вітку і обновлюємо робочу директорію до її стану, дозволяэ пересуватись по комітах
  • git branch [назва вітки] - створюємо нову вітку
  • git branch -d [назва вітки] - видаляємо вибрану вітку
  • git push - посилаємо всі зміни з локальної папки на комп*ютері в космос на віддаленний репозиторій, щоб там збереглось
  • git push [назва віддаленного репозиторію] [назва вітки] - те саме, тільки детальніше
  • git pull - загружаємо історію із віддаленного репозиторію на свій комп*ютер і об*єднуємо її із тим, що маємо
  • git merge [назва вітки] - вносить зміни із вибраної вітки в поточну вітку, допомагає з*єднувати вітки
  • git clone [url-посилання на репозиторій на гітхабі] - скачує репозиторій разом із всіма змінами

В цьому файлі.gitignore записуються назви файлів, які будуть ігноруватись (зазвичай це якісь системні файли)

  • git revert - ОХРАНА ОТМЄНА, відміняємо дії попереднього коміту, але не видаляємо його, а робимо новий із станом до помилкового коміту, типу "Антикоміт"
  • git reset - дозволяє обрати коміт, що був зроблений раніше і "забути" все що ми робили потім. Поки ще не дуже ясно, читала ось тут і ось тут
  • git cherry-pick - використовуємо, коли потрібно взяти зміни, створені якимось комітом, і застосувати їх заново у вигляді нового коміту поверх поточної вітки
  • git rebase - це автоматизованний cherry-pick, але для ланцюга комітів, наче переносимо вітку в інше місце: допомагає об*єднати зміни, які зроблені в одній гілці з іншою гілкою
  • git stash - фіксуємо стан нашої робочої директорії (змінені файли, які ми відслідковуємло і проіндексовані зміни) і зберігаємо їх в сховище незавершених змін, таким чином ми можемо в будь-який момент їх застовувати, зручно використовувати, коли ми хочемо змінити вітку при роботі і попрацювати над чимось іншим, при цьому не хочемо робити коміт?
